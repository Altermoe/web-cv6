## 1. 项目愿景与目标

### 1.1 终极目标
在浏览器中以**纯 WebGPU** 渲染管线为基石，复刻《文明 6》(Civilization VI) 的核心玩法与视觉体验，交付一个可玩、可扩展、跨平台的 Web 版本。

### 1.2 阶段性目标（M0 → M4）
| 里程碑 | 名称 | 核心交付 |
| --- | --- | --- |
| M0 | 基建期 | 仓库脚手架、CI、构建工具链、纯 WebGPU Hello-Triangle |
| M1 | 引擎原型 | 渲染器、资源管线、相机/输入/UI 基础框架、瓦片地图（无规则） |
| M2 | 地图与单位 | 六边形瓦片地形（平原/森林/山脉/海洋/河流）、迷雾、单位移动、视野 |
| M3 | 回合与城市 | 回合循环、食物/生产力/金币、研究、蛮族、城市建造与地块工作 |
| M4 | 完整一局 | 胜利条件（科技/文化/征服/宗教/外交）、AI 雏形、存档、设置菜单 |

> 注：每达成一个里程碑后进行回顾与计划调整；不追求一次性覆盖全部系统。

## 2. 技术栈选型

### 2.1 核心原则
- **混合架构（WebGPU + DOM）**：3D 场景层（地图 / 运动对象 / 特效）由纯 WebGPU 自研渲染；**平面 UI 全部由 DOM + 响应式框架承载**。两套体系通过明确的数据 / 事件契约解耦，各司其职。
- **纯 WebGPU**：3D 渲染层不依赖 WebGL / Three.js / Babylon.js 等高层引擎，渲染管线与 WGSL 着色器自行实现。
- **职责边界**：WebGPU 层只负责"看得见的 3D 世界"；UI 层不直接读写 GPU 资源，3D 层不创建 DOM 节点，严禁跨层调用。
- **可渐进增强**：当浏览器不支持 WebGPU 时给出明确提示，不做 WebGL 回退（保持栈的纯粹性）；此时 UI 层仍可降级为只读视图。
- **零运行时依赖最小化**：3D 核心层尽量只依赖浏览器原生 API；UI 层使用响应式框架，但其逻辑不得渗透到渲染层。

### 2.2 技术清单

| 类别 | 选型 | 用途 |
| --- | --- | --- |
| 语言 | **TypeScript (strict)** | 全栈逻辑 |
| 渲染 | **WebGPU API** + **WGSL** | 着色器、渲染管线 |
| 构建 | **Vite** | 开发服务器、HMR、产物打包 |
| 资源 | `glTF` / 自定义二进制 | 3D 模型、动画、纹理 |
| 数学 | `gl-matrix` | 向量/矩阵/四元数 |
| 状态 | **自研 ECS**（轻量） | 游戏世界、回合系统 |
| 持久化 | IndexedDB（`idb-keyval`） | 存档 |
| 测试 | Vitest + Playwright | 单元与端到端 |
| 工具链 | ESLint / Prettier / Husky / lint-staged | 代码质量 |
| 部署 | 静态站（GitHub Pages / Vercel） | 部署与展示 |

### 2.3 浏览器目标
- Chrome / Edge 113+（WebGPU 稳定通道）
- Firefox Nightly（启用 `dom.webgpu.enabled`）
- Safari Technology Preview（实验支持）
- 在不支持的浏览器中给出友好降级页（不渲染游戏）。

## 3. 总体架构

> 详细分层、monorepo 结构、3D core / UI / 后端依赖方向、Pinia 状态分层等工程化设计见 **[docs/architecture.md](docs/architecture.md)**。本节只列要点。

### 3.1 项目形态
- **monorepo(pnpm workspaces)**:3D 核心、UI、前端应用、后端、共享协议拆为独立 package。
- **3D core 与前端应用分包**:3D 核心(`packages/gpu3d`)对前端应用(`apps/web`)只暴露 `Renderer` facade / `InputPort` / `StateStream`,不泄漏 WebGPU API。
- **后端可选**:默认单机离线模式;检测到后端可达且用户启用时,才走联机/计算下沉。

### 3.2 混合渲染分层
```
┌──────────────────────┐
│  Frontend App (Vue3) │  ← 平面 UI + Pinia 状态
└──────────┬───────────┘
           │ 状态 / 事件契约 (bridge)
┌──────────┴───────────┐
│  3D Core (WebGPU)    │  ← 地图 / 单位 / 特效渲染
└──────────┬───────────┘
           │
┌──────────┴───────────┐
│  Backend(可选)       │  ← 计算下沉 / 多人同步
└──────────────────────┘
```

## 4. 渲染系统设计（WebGPU 核心）

### 4.1 渲染管线(仅 WebGPU 3D 部分)

> **UI 不属于 WebGPU 通道。** HUD / 菜单 / 模态框 / 设置页全部由 DOM + 响应式框架承载;3D 层只负责"看得见的 3D 世界"(地图、运动对象、特效)。
> 需要在 3D 空间中显示的 2D 元素(如小地图、战斗飘字)仍走 WebGPU 文本 / 图集 Pass。

1. **G-Buffer 通道**：地形 / 单位 / 城市分层渲染到 MRT 目标。
2. **光照通道**：方向光 + 阴影（单方向 ShadowMap，PCF 软阴影）。
3. **后处理通道**：色调映射、FXAA。
4. **3D 空间叠层通道**（可选）：小地图、伤害飘字、选中指示环。

### 4.2 关键渲染特性
- 单一 `RenderPass` 合并：地形/单位批处理（合批 + 实例化）。
- **实例化渲染**：六边形地块、网格单位使用 `instance_index` 着色。
- 着色器：统一 UBO（相机/光照/时间）跨 Pass 共享。
- 纹理压缩：BC / ETC2，运行时检测设备能力。
- LOD：模型 / 地形网格按相机距离分级。

### 4.3 着色器规范
- 入口：`@vertex` / `@fragment` 显式声明。
- Uniform：`CameraUBO`、`LightUBO`、`TerrainUBO`、`TimeUBO`。
- 数据结构：地形属性压缩为 RGBA8（高度 / 生物群落 / 湿度 / 通行代价）。

### 4.4 渲染层 ↔ UI 层契约
- **渲染层向上暴露**：
  - `InputPort`：接收标准化输入（点击坐标、拖拽、键盘）。
  - `StateStream`：可订阅的世界状态（选中单位、回合数、资源、相机位姿）。
- **UI 层向下发送**：
  - `Command`：玩家操作（结束回合、暂停、存档、设置变更）。
  - `InputDispatch`：把 DOM 事件透传给 3D 层。
- **禁止**：
  - 3D 层调用任何 UI 组件 / DOM API。
  - UI 层直接访问 WebGPU 设备或着色器。
- **坐标系**：`ui-plane`（屏幕像素）与 `world-plane`（世界坐标）互不混用；通过 bridge 层的纯函数 `screenToWorld` / `worldToScreen` 转换。

## 5. 游戏系统设计

### 5.1 地图系统
- **六边形网格（Axial 坐标）**，支持 30/36/42/48/60 大小的标准与巨大地图。
- 地形类型：海洋、湖泊、草原、平原、森林、雨林、沙漠、丘陵、山脉、雪原。
- 地形生成：分形噪声 + 气候模拟（纬度温度 / 经度湿度）+ 河流沿最低点。
- 资源 / 奢侈品 / 战略资源散布。
- 迷雾：可视、已探索、不可见三态（位图存储于 GPU 纹理）。

### 5.2 回合与时间
- 顺序：玩家回合 → AI 回合 → 下一回合。
- 节奏：`setTimeout(0)` 切分 yield，避免长任务阻塞。
- 事件驱动：每回合消费时间轴事件队列。

### 5.3 单位与移动
- 单位组件：类型、阵营、位置、HP、行动力、视野、经验、升级。
- 移动规则：按地形消耗行动力；ZOC（控制区）阻止穿越。
- 战斗：近战 / 远程 / 攻城 / 海军四种基本类型 + 地形 / 科技加成。

### 5.4 城市系统
- 城市地块：环形 6 半径。
- 产出：食物 → 人口；生产力 → 建筑 / 区域 / 单位。
- 区域：学院、商业、圣地、剧院、军营、工业、港口、宇航中心等（按里程碑逐步加入）。
- 满意度：每城独立计算，受奢侈品 / 政府 / 战争影响。

### 5.5 科技与文化
- **科技树**：前向图，按里程碑裁剪（先做 M3 必需要素）。
- **文化树**：社会制度（古/古典/中古/近代/现代/信息），政策槽。
- 关键效果：单位解锁、建筑解锁、产能加成。

### 5.6 胜利条件（M4）
- 科技胜利（送卫星 / 火星殖民地）
- 文化胜利（境外游客）
- 征服胜利（占领所有首都）
- 宗教胜利（单一宗教主导）
- 外交胜利（紧急状态投票）

### 5.7 AI（M4 雏形）
- 城市级：效用 AI 决定建什么。
- 军事级：行为树（探索 → 扩张 → 防守 → 进攻）。
- 战略级：长程目标 + 威胁评估。

## 6. UI / UX

> **核心原则**:UI 全部由 **DOM + 响应式框架** 承载,**不进入 WebGPU 通道**;3D 层只负责地图 / 单位 / 特效的渲染,UI 与 3D 通过 `bridge/` 下的事件 / store 契约通信。

### 6.1 架构边界
- 响应式框架默认 **Vue 3 + `<script setup>` + Pinia**(候选:Svelte 5 / Solid.js,详见 §2.2)。
- UI 与 3D 层通过 **事件总线 + 状态切片(store)** 解耦;3D 层只暴露:
  - `InputPort`:接收标准化输入(点击坐标、拖拽、键盘)。
  - `StateStream`:可订阅的世界状态(选中单位、回合数、资源、相机位姿)。
- 3D 层 **不直接调用** 任何 UI 组件 / DOM API;UI 也 **不直接读写** GPU 资源(见 §4.4)。

### 6.2 页面与 HUD
- 页面:主菜单 → 单人游戏 → 设置 → 加载存档。
- HUD:回合数、金币 / 科技 / 文化 / 信仰 / 旅游;左下单位信息;右下小地图(由 WebGPU 3D 通道渲染,但被 UI 层通过 store 控制其位置 / 显隐 / 缩放)。
- HUD **位置、层级、可隐藏性** 通过响应式 store 控制,DOM 浮层通过 CSS `pointer-events` 与 3D 画布的事件分发协调。

### 6.3 输入
- 鼠标左键:选择 / 移动 / 攻击。
- 鼠标右键:移动命令。
- 滚轮:缩放;Q/E:旋转;WASD:平移。
- Space:结束回合;Esc:暂停。
- 触屏:长按 = 右键,捏合 = 缩放。
- **输入路由**:UI 层先接收 DOM 事件 → 命中 HUD 元素则由 UI 消费 → 否则透传给 3D 层的 `InputPort`(命令模式 + 选择模式切换)。

### 6.4 国际化与可访问性
- 国际化:i18n(中 / 英),文本文案集中于 `i18n/*.json`。
- 全部 UI 元素需满足键盘可访问与 ARIA 标注;HUD 文本支持字体缩放。

## 8. 风险与对策

| 风险 | 影响 | 对策 |
| --- | --- | --- |
| WebGPU 浏览器支持有限 | 覆盖面 | 明确不支持提示 + 引导升级；不投入 WebGL 回退 |
| Civ6 版权 / 美术资产 | 法务 | 不直接使用官方美术；以 CC0 / 自制资源替代 |
| 六边形地图 + 阴影 / 后处理性能 | 帧率 | 实例化 + 渲染图裁剪 + 动态分辨率 |
| 规则系统庞杂 | 工期 | 严格按里程碑分批；先骨架后细节 |
| 存档兼容性 | 用户体验 | 版本号 + 迁移器 + 失败降级 |
| AI 工作量爆炸 | 工期 | M3 之前使用脚本化对手即可 |

## 9. 开发流程

1. **Git Flow 简化**：`main`（可发布）+ `feat/*`（特性）+ `fix/*`（修复）+ `chore/*`（杂项）。
2. **提交规范**：Conventional Commits，自动生成 CHANGELOG。
3. **CI（GitHub Actions）**：lint → typecheck → unit test → build → e2e（Playwright）。
4. **评审**：每个里程碑产出 `docs/milestones/M{n}.md` 总结并复盘。
5. **每周一次回顾**：进度、风险、下一阶段切片。
